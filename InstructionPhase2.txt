Lab Session: Knowledge Base Construction,
Alignment, and Expansion
(Preparation for Knowledge Graph Embedding)
Objective
In this lab, you will construct a private Knowledge Base (KB), align it with an open Knowledge Base
(DBpedia or Wikidata), extend it via SPARQL queries, and prepare it for the next lab session on Knowledge
Graph Embedding (KGE).
The final KB must be large enough to support meaningful embedding learning while remaining
computationally manageable on a laptop.
An example toy data is provided for better explaination in each step.
Step 1 --- Build the Initial Private Knowledge Base
Requirements
You should have prepared your own knowledge graph in RDF model, with data collected from some web API
or html pages. It is adviced that your starting knowledge graph respects the following volume:
≥ 100 triplets
≥ 50 entities
No upper limit
Example (Domain: Scientific Researchers)
:MarieCurie :wonAward :NobelPrize .
:MarieCurie :fieldOfWork :Physics .
:AlbertEinstein :fieldOfWork :Physics .
:AlbertEinstein :wonAward :NobelPrize .
:RosalindFranklin :affiliatedWith :KingsCollegeLondon .
Good Practices
Use URIs, not plain strings
Normalize predicate naming (e.g., camelCase)
Avoid duplicate triples
Clearly separate entities from literals

Step 3 --- Predicate Alignment Using SPARQL
Requirement
Use SPARQL queries on the open KB to determine whether:
An identical predicate exists
A semantically similar predicate exists
Validation can be manual. Here is a simple way: Given a triplet $(s,p,o)$ in your private knowledge graph,
the subject $s_{1}$ and object $o_{1}$ are alighed with {wiki(s)} and {wiki(o)} in the open KB, you can list all
existing prdicates returned by the SPQRAL Query
SELECT ?property ?propertyLabel WHERE {
 ?property a wikibase:Property .
 ?property rdfs:label ?propertyLabel .
 {wiki(s)} ?property {wiki(o)}.
}
Then manually compare the results with $p$ in your private knowledge graph, chose the most pertinent one
for alignement.
Of course other more intelligent ways exist. You will learn later.

Step 4 --- KB Expansion via SPARQL
Final Size Requirements
After expansion, your KB must contain:
50,000 -- 200,000 triplets
5,000 -- 30,000 entities
50 -- 200 relations
If the KB is too small → KGE models will fail to learn meaningful structure.
If too large → computation may exceed laptop capabilities.
Anchored Expansion Strategy
Only expand from confidently aligned entities.
1-Hop Expansion
SELECT ?p ?o WHERE {
 wd:Q7186 ?p ?o .
}
LIMIT 1000

Cleaning Before Embedding
Before exporting for KGE:
Remove duplicate triples
Remove inconsistent URIs
Ensure consistent predicate alignment
Remove unwanted literal-heavy predicates (if necessary)
Ensure graph connectivity (avoid isolated components)
Why These Volume Constraints Matter
KB Size Consequence
< 10k triples Embeddings collapse to frequency effects 50k--200k triples Stable training and meaningful
structure > 500k triples High memory and computation cost
